# WORKLOG 2026-07-23

## 今日やったこと：filovita-mvp — 旅の作り直し・森の不具合修正・手帳の実用文具・Google連携の方針決定

### 1. 森（ForestGrowth）の重なり不具合を修正

設定画面で入力方法や暮らしのテーマを選ぶボタン類が、背景の木や木漏れ日の下に隠れて見えなくなっていた。`ForestGrowth.jsx`の`children`ラッパーが`position:relative`を持たない非配置要素のままだったため、後続の絶対配置レイヤー（木・木漏れ日オーバーレイ等）がz-indexに関わらず前面に来てしまっていた。`position:relative; zIndex:1`を明示して解決。

### 2. 旅（旧・冒険）の再設計 — 2段階

**1段階目**：地図が5段階で描き足られていく方式で実装・デプロイ（`travel-map-0〜4.png`、押し花/チケット/スタンプの記録あと）。

**2段階目（ユーザーからの根本的な指摘で作り直し）**：
- 「絵が小さな枠に押し込まれて勿体ない、SF・ホラーみたいに背景いっぱいに重ねられないか」との指摘
- 調査の結果、旅の素材は「日記帳に貼るパーツ集」（小さな地図カード＋ステッカー群、解像度も低め）であり、SF/ホラーの一枚背景とは性質が違うことが判明。正直に説明し、方針を相談
- ユーザーが「縦長の完成イメージ画像（`travel_scene.png`）をそのまま背景に使ってほしい」「地図・帳面は固定のまま、記録が増えるほどスクラップの印が増えていく」という方向に整理
- 実装：`travel_scene.png`から偽のアプリUI部分（上部ヘッダー・下部ナビ）を除いた本体イラストを`travel-journal-bg.jpg`として書き出し、`JournalMap.jsx`をSF/ホラーと同じ「App.jsx直下のフルブリード・コンポーネントテーマ」に作り替え
- 背景全体に薄いクリーム色のウォッシュ（`rgba(250,244,230,0.55)`）を重ね、実際のUI文字がどこに乗っても読めるようにした
- しるし（押し花/チケット/スタンプ）は「最後まで全部埋めない・紙面の余白にだけ置く・重ならない」の3原則で、8つの固定枠×最大6つ表示・日付ハッシュで衝突回避、という配置ルールに作り直した
- 旅ノートだけ文字が弱く見える指摘を受け、Klee One（手書き風フォント）を`bodyFont`として導入（InputScreen・ConfirmScreen・EventDetailScreenの所感欄）
- インク色を茶色系からブルーブラック（万年筆色、`#1C2B45`系）に変更

### 3. 「テーマ＝生活スタイル」という発想の展開 → 手帳テーマに実用文具を追加

ユーザーから「テーマによって機能まで変わってよい、手帳だけは実物の手帳文化を再現した“プロフェッショナル版テーマ”にできる」という提案があり、以下を実装：

- **チェック**：手帳らしいボールペン風チェックマーク（`PlannerCheck.jsx`）
- **インデックス**：タグを「#医療」ではなく「📑 医療」表示に（`IndexTab.jsx`）
- **スタンプ**：済・延期・電話・病院・買い物・大事の6種。タップでtagsに追加/削除するだけで、データ構造は増やしていない（`StampBadge.jsx` / `stamps.js`）
- **ページリンク**：次回予定・関連Eventを「→ 8月5日を見る」のようにクリック可能にし、実際にその日のページへ飛べる（`pageLink.js`）。副次的に`formatDateLabel()`が月を無視して常に「7月」と表示していた既存バグを発見・修正
- **タグの道具箱**：「タグは分類ではなく、自分専用の道具箱である」というユーザーの発見に基づき実装。📑タグをタップすると、そのタグ専用の参照リスト（Web/地図/電話/ファイル/Event）を開いて追加・編集・削除できる。`tagRegistry`（タグ名→tagId）と`tagToolboxes`（tagId→参照リスト）を新設し、タグ名ではなく内部IDで紐付けることで、将来タグの表示名を変えても道具箱との対応が壊れないようにした（ユーザー自身の設計指摘）
- **強調機能（マーカー等）**：「今回の結論」の文章を選択すると、🟨マーカー/✏️アンダーライン/⭐重要/❗注意/📌あとで見る の5つが選べるポップアップが浮かぶ。選ぶとその一文だけに印がつき、タップで解除できる（`MarkableText.jsx` / `marks.js`）。タグの道具箱には「マーカーされた記録」欄を追加し、そのタグがついた記録の中でマーカーされた一文だけを日付付きで横断的に一覧表示する仕組みも実装
- マーカーの帯を、ユーザーの指摘（「紙の蛍光ペンは文字の上下に少し余白が残る」）を受けてlinear-gradientの帯に変更。%指定だと改行で折り返した行ごとに帯の高さがずれる不具合があったため、em単位の固定値に直した

いずれも他のテーマ（森・ホラー・SF・サイバーパンク等）には影響しないよう`isTecho`で分岐しており、都度スクリーンショットで regression チェック済み。

### 4. Google Calendar連携の方針が固まった（実装はまだ）

ユーザーから「Filovitaはデモなのか、実用段階なのか」と問われ、現状を正直に説明：
- チェック・タグ・スタンプ・マーカー・道具箱などの操作自体は本物（localStorage永続化）
- ただしAI下書き生成は偽物（`fakeGenerateDraft.js`）、カレンダー連携は**UIのみでGoogle Calendar API・OAuth・データ構造は一切未着手**（`CalendarConnectScreen`の「連携する」「あとで設定する」は同じ処理を呼ぶだけ）、今日の日付も固定、保存先もブラウザのlocalStorageのみ

この説明をきっかけに、Google Calendar連携の設計方針が固まった：

- **同期ではなく「取り込み」**：Googleは予定の管理者、Filovitaは生活記録の管理者、と役割を分離する
- 再取り込み時はGoogle由来のフィールド（日時・件名など）だけ更新し、Filovita側で育てたメモ・タグ道具箱・マーカー・関連リンクなどは絶対に上書きしない
- Eventに`source`・`googleEventId`のような印を持たせ、マージ規則を明確にする設計
- ユーザーの判断：「もうモックを増やす段階ではない」——手帳・タグ道具箱・マーカーなど中身がかなり育った今、入口（カレンダー連携）だけ偽物のままだと体験が「工作」っぽくなる。本物の生活と繋がって初めて面白くなる段階に来ている、との認識
- 次にやる範囲を明確化：Google認証→カレンダー一覧取得→期間指定して予定読み込み→Filovita Eventとして保存→再取り込み時はGoogle由来部分だけ更新、Filovita側の追記は保持。双方向同期・繰り返し予定の完全対応・参加者編集などの重い部分は後回し
- **今回は方針決定のみ。実装は次回（明日）に着手する**、とユーザーから明示された

なし。ただし森の重なり不具合（設定画面のボタンが見えなくなっていた）は、以前ホラー・SF・旅で同種のz-index対応をした際に森だけ見落としていたもの。新しいコンポーネントテーマを作るたびに「childrenラッパーにposition:relative + zIndex:1があるか」を確認するチェック項目として明示しておく。

## メモ・気づき

- 実イラスト素材を使う際は、「1枚の連続した背景シーン」なのか「小さな部品を組み合わせるパーツ集」なのかを、解像度と構図から早めに見極める必要がある（旅の1段階目はこれを見誤った）。
- 「データ構造は増やさない機能」（チェック・インデックス・スタンプ・ページリンク）と「新しいデータ構造が要る機能」（タグ道具箱・マーカー）は、実装コストも設計判断の重さも別物として扱うとよい。
