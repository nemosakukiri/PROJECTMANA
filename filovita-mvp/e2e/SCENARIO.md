# E2E シナリオ：予約票の撮影から、後日その情報で支援を受けるまで

本番を想定した「生活の中で最後まで目的を達成できるか」を検証するための、
Filovitaの一連の利用フロー。単体機能ではなく、一人の利用者が実際に辿る
時系列で記述する。

**このファイルは品質確認のための資料であると同時に、設計書でもある。**
Filovitaでは「利用者がこう使う」というシナリオが、そのまま「AIはどう
支援すべきか」という仕様になる——設計書とテストが別物として存在するの
ではなく、この一本のシナリオが両方を兼ねる。ステップを追加・変更する
ときは、テストの都合ではなく「実際の生活でどう使われるか」を基準に
書くこと。

各ステップに実装状況を明記する。`full-life-journey.spec.mjs` は
**実装済みの部分だけ**を、本物のUI操作（クリック・入力・リロード）を
通して連続実行する。実装されていない部分はテストでは飛ばし、その旨を
このファイルとテスト実行時のログの両方に明記する。

## 今回のテストが検証すること

1. 音声入力（モック）→ AI下書き → 確認 → Event確定、が実際にカレンダーへ反映される
2. 所感を書くと`所感`タグが自動で付き、タグの道具箱を開いて参照（電話番号）を登録できる
3. 登録した参照から「電話する」ボタンが正しい`tel:`リンクを持つ
4. 結論文にマーカーを引ける
5. **ページを実際にリロードしてもすべて残っている**（Event・タグ・道具箱の参照・マーカー）
6. リロード後、タグの道具箱を開くと「マーカーされた記録」欄に、後で見つけたマーカーが日付付きで一覧表示される
7. 他のテーマ（このテスト対象外のテーマ）を壊していないかは、このテストの範囲外。回帰確認は都度Playwrightでスクリーンショットを撮って手動で見比べる運用を続ける（別途スクリプト化はまだしていない）

## 未実装ギャップ（次に埋めるべきもの、優先順）

1. Google Calendar実連携（すでにWORKLOGに記録済み、次回セッション最優先）
2. 予約票・チラシの撮影 → OCR → AI抽出のパイプライン
3. タグの自動生成（現状はスタンプの手動選択と「所感」のみ。写真からのAI抽出ができて初めて`#病院`のような意味のあるタグが自動で付く）

## 運用ルール：完成品ではなく、生きたシナリオとして育てる

追記日: 2026-07-23

このE2Eは一度作って終わりのテストではなく、**Filovitaの品質保証の基盤**として、
機能追加のたびに育てていく。

- **新しい機能を実装したら、その場で`full-life-journey.spec.mjs`にステップを足す。**
  「あとでテストを書く」にしない——今日のセッションで実装した機能はすべて、
  実装したその流れの中で組み込んだ（音声入力・タグの道具箱・マーカー・電話
  するボタン、すべて同じテスト内で連続検証している）。
- **未実装のステップは、消さずに`gap()`として明示したまま残す。** 実装したら
  `gap()`を実際のステップに置き換える。上の実装状況の表も同時に更新する。
- **最終目標**：「予約票を撮影する」→「OCR・AIが読み取る」→「Eventになる」→
  「カレンダーに載る」→「AIの長期記憶に育つ」→「後日、その情報でAIが支援を
  完了する（電話をかける・地図を開くところまで）」が、**モックなしで一本の
  流れとして通ること**。今はまだ音声入力・確認・登録・道具箱・マーカーの
  部分だけがモックなしで通っている状態。写真撮影・OCR・Google連携が実装
  されるたびに、この一本のテストの中の`gap()`が減っていく。
- Google Calendar連携（次回セッション）を実装したら、真っ先にこのシナリオに
  組み込むこと。「⑤ Eventがカレンダーに登録される」のGoogle側部分と、
  「① 予約票を撮影する」「② OCR・AI解析」を、モックに頼らず実装した分だけ
  このテストで置き換えていく。
