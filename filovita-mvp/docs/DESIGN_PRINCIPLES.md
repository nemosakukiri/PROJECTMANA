## なぜテーマを作るのか

Filovitaのテーマは、見た目を変えるためのものではない。

それぞれのテーマは、「時間との付き合い方」という問いに対する、一つの世界からの答えである。

利用者はテーマを選ぶのではない。その月をどの世界で暮らすかを選ぶ。

---

## Theme Design Questions（世界を設計するときの問い）

テーマを設計するとき、最初に考えるのはUIではない。

最初に考えるのは、その世界では時間がどのように流れるかである。

- この世界では時間は何を変えるのか。
- 利用者は時間を操作するのか、それとも立ち会うのか。
- 変化は価値なのか、安心なのか。
- 一か月はどのような物語になるのか。
- 利用者は何を持ち帰るのか。

UI・画面構成・データ構造は、これらの問いに対するその世界なりの答えが固まったあとで初めて検討する。

新しいテーマ(ホラー・絵本・SF・サイバーパンクなど)を作るときも、森の6項目をそのまま言葉だけ入れ替えて流用してはならない。それは思想の使い回しではなく、装飾の使い回しである。同じ問いに対して、その世界自身の答えを新たに見つけること。

## もう一つの問い：月末に、どんな一枚の絵が完成するのか

追記日: 2026-07-23
経緯: SFの実装中、テーマ設計者(illustrator含む)にとって「時間との付き合い方」という問いは正しいが抽象的すぎる、という指摘から見つかった、もう一つの言い方。

**「時間をどう感じるか」は思想としては正確だが、絵を描く人には伝わりにくい。** もっと具体的で、誰とでも同じゴールを共有できる問いがある。

> その月の終わりに、どんな一枚の絵が完成するのか。

これは「時間をどう感じるか」の代わりではなく、それを絵に翻訳したときの答えである。テーマ設計書は、この一行だけでも成立する。

| テーマ | 完成する一枚の絵 |
|---|---|
| 森 | 一か月をかけて育ち切った、夏の森 |
| ホラー | 暮らしの跡が積み重なった、誰かが住んでいる館 |
| 絵本 | 花・風船・夏まつりの灯りで埋まった、絵本の一ページ |
| SF | 窓いっぱいに広がる目的地の惑星と、観測ログが増えたコックピット |
| 旅 | 地図・押し花・切符・スタンプが散りばめられた、旅ノートの見開き |

この観点に立つと、Filovitaは「背景が変わるアプリ」ではなく、**毎月、自分だけの一枚の作品を静かに完成させていくアプリ**になる。月末に一望する体験(森の設計時に見つけた「変わらない日にも価値がある」の延長)は、この一枚の絵を眺める時間そのものである。

### 問いをもう一段抽象化する：「完成」ではなく「姿」

追記日: 2026-07-23
経緯: サイバーパンクのTime Philosophy(「安定した完成形に向かわない、常に上書きされ続ける」)は、「月末に一枚の絵が完成する」という言い方とは馴染まない。無理に例外扱いするのではなく、問い自体を一段抽象化することでこれを解決する。

> その月の終わりに、どんな**姿**が存在するのか。

「完成する」を「姿がある」に置き換えると、完成に向かうテーマも、更新され続けて終わらないテーマも、同じ問いの中に自然に収まる。

| テーマ | 月末の姿 |
|---|---|
| 森 | 育った姿 |
| 旅 | 記録された姿 |
| SF | 到達直前の姿 |
| ホラー | 住み慣れた姿 |
| サイバーパンク | 更新され続ける姿 |

WindowsもiOSもLinuxも完成しない。アップデートされ続ける。だから月末は「Version 7.31」というだけ——それ自体が、その瞬間の姿である。サイバーパンクにとっての「月末の一枚」は、完成した絵ではなく、バージョン番号のようなスナップショットでよい。この一段の抽象化により、サイバーパンクはもう例外ではない。

## 森という答え（実例）

上記の問いに対して、森の世界が出した答えは以下の6つである。

1. 時間だけが森を育てる。利用者の操作では育たない。
2. 利用者は森を育てない。暮らした痕跡だけを残す。
3. 変化は希少である。毎日何か起きる必要はない。
4. 変わらない日にも価値がある。同じ景色は安心として設計する。
5. 月には固有の物語がある。7月と10月は別の世界。
6. 森は終わらない。月が終わっても、一冊の絵本として残る。

これは森というテーマの仕様であって、Filovita全体の仕様ではない。他のテーマは、この6項目とは違う答えを出してよい。むしろ出すべきである。

## World Engine（実装骨格）

追記日: 2026-07-23
経緯: 森の実装を通じて見つかった、テーマ共通の骨格。森を作るエンジンではなく、
「一か月を一つの世界として暮らすためのエンジン」として整理し直したもの。
Filovitaはテーマを追加するアプリではなく、世界を追加できるプラットフォームである、
という理解に到達したのはこの整理があったからである。

どの世界も、以下の6つを持つ。中身(絵・言葉・頻度)はテーマごとにまったく違ってよい。骨格だけが共通。

```
World
├─ Time Philosophy   時間は何を変えるのか（森は5段階の育ち方）
├─ Monthly Story     その月固有の物語（7月なら蛍・梅雨明け）
├─ Living Changes    説明できない小さな今日らしさの積み重ね
├─ User Traces       記録した日にだけ残る、暮らしの跡
├─ Seasonal Tone     季節としての色・空気（今は Monthly Story に内包）
└─ Rare Moments      Living Changesよりさらに希少で、目立ってよい特別な一日
```

実装は `theme/worldEngine.js` にある。テーマ非依存の部分（日付計算、信号のシード生成、痕跡の配置）だけがここに置かれ、絵や物語は各テーマ側(`theme/<name>/`)が持つ。

### Living Changes ≠ Rare Moments（重要な区別）

最初の実装は「5日に1回、倒木という"発見イベント"が起きる」というものだった。これは間違いではないが、機械的だった。

**Rare Moments（イベント）** は、「起きたか起きていないか」がはっきりした特別な一日を作る。目立ってよい。

**Living Changes（生活）** は、それとは違う。蝶がいる、朝露が残っている、鳥の気配がする、水面に波紋がある、木漏れ日の位置が違う——これらは独立した小さな信号で、同時に起きても、何も起きなくてもいい。一つの合図に頼らず、複数の小さな変数が重なることで「なぜかは分からないけど今日は違う」が生まれる。これが「世界が生きている」ということであり、「発見を作る」こととは別物である。

森の実装では両方を残した。Rare Moments(倒木)は希少に、はっきりと。Living Changes(蝶・朝露・鳥・波紋・木漏れ日の位置)は、それぞれ独立した頻度で、日常的に。

### Time Philosophyは共通化しない

Living Changes / Rare Moments / User Tracesは仕組み（シード付き信号、痕跡の配置）を共有できる。しかしTime Philosophyは違う。「時間が何を変えるのか」という問いは共通でも、その答えの**構造そのもの**がテーマごとに違ってよい。

- 森：時間は森を育てる（世界が変わる。積み上がる方向）
- ホラー：時間は恐怖を親しみに変える（利用者の感じ方が変わる。世界はほぼ変わらない）
- 絵本：時間はページをめくる
- SF：時間は目的地までの距離を縮める
- 旅：時間は地図を描き足す
- サイバーパンク：時間は世界を書き換え続ける（安定した完成形に向かわない。「月末の姿」参照）

森とホラーは、どちらも「stage 0〜4」という同じ日付区切りを使っていても、中身は正反対である。森は主役が世界（景色が変わる）。ホラーは主役が感情（同じ景色の受け取り方が変わる）。だからTime Philosophyを実装するとき、「前のテーマの仕組みを流用してから中身を変える」のではなく、まず「主役は世界か、感情か、それとも別の何かか」を先に決める。

## ホラーという答え（実例）

上記の問いに対して、ホラーの世界が出した答えは以下である。

1. 時間は部屋を変えない。部屋は最初からずっと同じ——変わるのは利用者の感じ方だけ。
2. 恐怖は親しみに変わっていく。「怖い」→「暖炉の火を長く見てしまう」→「静かだな」→「もう馴染んだ」という一本の弧。
3. 世界ではなく感情が主役。森が「景色が変わる」なら、ホラーは「同じ景色の受け取り方が変わる」。
4. 希少な出来事は説明しない。いつも閉まっている扉が、その日だけ少し開いている。それだけで、開ける必要も理由もない。
5. 利用者が残す跡は、暮らし始めた証。栞・引かれた椅子・置かれたカップ——恐怖の館ではなく、住み始めた館。

これも森と同じくホラーというテーマの仕様であって、Filovita全体の仕様ではない。

## 絵本という答え（実例）

絵本にとって、時間は世界を育てもしないし、感じ方も変えない。ページをめくるだけである。7月の絵本は「とびらをめくったところ→麦畑の道→夏まつり(いちばん賑やかな見開き)→長い影の帰り道→おうちに灯りがともる」という、始まり・山場・終わりのある一つの物語になる。

1. 時間はページをめくる。世界も感情も主役ではなく、物語の進み具合が主役。
2. 一か月は起承転結のある一つの話になる。森のような一方向の蓄積ではなく、山場(夏まつり)のあと収束していく弧を持つ。
3. 希少な出来事は、ページの中に隠れている小さな生き物。絵本の「かくれんぼ」の作法にならい、説明も解決もしない。
4. 利用者が残す跡は、そのページに挟んだ押し花のような、小さな印。
5. カレンダーは絵本の外に貼り付けた道具ではなく、見開きの「文字」の側にある。だから絵の下にそのままの大きさで置く——挿絵に気を取られて実用性を犠牲にしない。

これも森・ホラーと同じく絵本というテーマの仕様であって、Filovita全体の仕様ではない。

## サイバーパンクという答え（実例）

サイバーパンクはUI(CRT・鋼板・リベット)がすでに高い完成度だったため、Time Philosophyだけを足した。時間は世界を育てず、感じ方も変えず、ページもめくらない。ただ**稼働し続け、書き換わり続ける**。安定した完成形に向かわない。

1. 時間は世界を書き換え続ける。森・ホラー・絵本のように、ある終着点(満ちた森、馴染んだ家、閉じた物語)に向かわない。ログが積み上がりながら、常に最新の状態に上書きされる。
2. 一か月は物語ではなく、稼働記録(BOOT→SIGNAL WEAK→SYNCING→STABLE LINK→UPTIME 30D)。他のテーマが持つ「始まりと終わり」がない。
3. 希少な出来事は、紛れ込む素性の分からないプロセス名。説明はしない。
4. 日々のLiving Changesは、信号強度のようなノイズ。意味を持たせない、ただの揺らぎ。
5. 利用者が残す跡は、接続済みのノード。暮らしの跡でありながら、ネットワークの語彙で語られる。

これも他テーマと同じくサイバーパンクというテーマの仕様であって、Filovita全体の仕様ではない。

## SFという答え（実例）

SFにとって時間は、世界を育てず、感じ方も変えず、ページもめくらない。ただ**目的地までの距離を縮める**。観測窓の向こうの惑星(Kepler-186f)そのものは同じ写真のまま——動くのは見出し下のキャプションに出る距離と進行度の数字だけ。完成する一枚の絵は「窓いっぱいに広がる目的地の惑星と、観測ログが増えたコックピット」。

1. 時間は距離を縮める。世界(森)でも感情(ホラー)でも物語(絵本)でもなく、軌道という即物的な進行が主役。
2. 一か月は「遠い星→少し近づく→表面の模様が見える→リングや衛星が見える→到着まぢか」という接近の弧になる。
3. 希少な出来事は、まれに映り込む素性の分からない飛行体(UFO)。説明はしない。
4. 利用者が残す跡は、記録した日にだけ残る観測ログの一行(流れ星を確認、小惑星帯を通過、など)。
5. 写真自体にDESTINATION/SYSTEM STATUSパネルが写り込んでいるため、同じ情報を重複してUIオーバーレイで重ねない——写真が語れることは写真に任せる。

これも他テーマと同じくSFというテーマの仕様であって、Filovita全体の仕様ではない。

## 旅という答え（実例）

追記日: 2026-07-23
「冒険」(RPGのクエスト・攻略言葉)を廃止し、「旅」として再設計した。旅にとって時間は攻略ではない。地図が少しずつ描き足されていく。森が「育つ」なら、旅は「描かれていく」——歩いた分だけ、まっさらな地図に道が伸びる。完成する一枚の絵は、その月だけの旅ノートの見開きページ。攻略ではなく作品。

1. 時間は地図を描き足す。世界(森)でも感情(ホラー)でも距離(SF)でもなく、「まだ何も描かれていない地図」→「森の小道を歩きはじめたところ」→「湖が見えてきたところ」→「高台からの景色が見えてきたところ」→「旅の全貌が見えてきたところ」という、白紙から満ちるまでの弧が主役。
2. 一か月は攻略のゴールに向かわない。クリアや達成ではなく、記録した分だけ地図が埋まっていく——利用者が「勝つ」テーマではない。
3. 希少な出来事は、まれに見つかる特別な発見(流れ星)。説明はしない。
4. 利用者が残す跡は、記録した日にだけ地図の余白に増えていく旅のしるし——きれいな花を見つけた、チケットを貼った、スタンプを押した。
5. 月末の姿は「記録された姿」——攻略し切った達成ではなく、歩いた分だけ残った旅ノートの見開き。

これも他テーマと同じく旅というテーマの仕様であって、Filovita全体の仕様ではない。

## 誰にも見られないから、自分の世界を選べる

追記日: 2026-07-23

紙の手帳は人の目に触れる。会議で机に置く、カフェで開く、電車で見る。だから無意識に「子どもっぽすぎないか」「仕事なんだから落ち着いたデザインを」と、自分の好みより他人の視線を優先して選んでしまうことがある。

Filovitaはスマートフォンの中だけの世界である。だから、50代の人が絵本を選んでもいい。サラリーマンがSF基地を育ててもいい。医師が森を眺めてもいい。誰もその世界観を評価しない。

これはFilovitaの装飾上の特徴ではなく、このアプリを開く楽しさそのものを支える前提である。テーマは「UI」ではなく「体験」であり、テーマを選ぶことは「今日、自分がどの世界に住みたいか」を選ぶことに等しい。

## 横に広げるより、縦に深める

追記日: 2026-07-23
経緯: 「テーマごとに専用の道具(木製のしおり、ホログラムマーカー等)を全部作ろう」という発想から、一日の議論の末にたどり着いた判断基準の転換。

良いプロダクトは、ある時期から「何を作るか」より「何を作らないか」の方が難しくなる。Filovitaは、テーマという名の世界観をすでに複数持っている(森・ホラー・絵本・SF・旅・サイバーパンク・手帳)。この状態で問うべきは「テーマをもっと増やせるか」「道具をテーマごとに全部作れるか」ではない。

> 本当にその世界に住んでいる気がするか。

森を開いた瞬間に「ああ、この世界に帰ってきた」と思えるかどうかは、マーカーが葉っぱの形をしているかどうかより、はるかに大きな価値を持つ。すでにTime Philosophy・Living Changes・Rare Moments・User Tracesという骨格がある以上、テーマごとの道具のバリエーションを増やしても体験の深さは増えず、選択肢と操作だけが増える。それは自由ではなく複雑さである。

だから今のFilovitaに必要なのは「世界観の実装競争」ではなく、毎日使う核——Google連携・Eventの育ち方・AIとの自然な関わり——を磨き込むことである。**この機能は本当に生活を豊かにするか**、という問いを基準にする。「機能を削る」のではなく「Filovitaらしくない機能は足さない」という判断を、意識して行う。

## タグは分類ではなく、書き込みを減らす仕組み

追記日: 2026-07-23

紙の手帳が付箋・インデックス・色分け・シールを使うのは、紙には検索もタグも関連付けもないからである。Filovitaはデジタルであり、タグ・検索・関連Event・タグの道具箱をすでに持っている。だから紙の文化をそのままデジタルに持ち込む必要はない——持ち込めば、デジタルなのに紙の不便さを再現してしまう。

`#病院`というタグを付けるだけで、病院のサイト・地図・関連Event・よく使う参照が呼び出せるなら、それは「付箋を貼らなくて済む」「インデックスを付けなくて済む」ということである。手帳好きな人ほど、同じ手帳でも書き方は人それぞれ違う。だから「こう使ってください」という道具を増やすより、余白を残す方がFilovitaらしい。

**手帳テーマは、紙の手帳を忠実に再現することではない。「手帳を使う人が心地よく感じる世界観」で十分である。** そこに最低限、マーカー・チェック・タグがあれば、あとは利用者自身が自由に使う。Filovitaは紙を再現するアプリではなく、デジタルだからこそ紙よりシンプルにできる部分を活かすアプリである。

## AIは先生ではなく、最初だけ隣にいる人（補助輪の原則）

追記日: 2026-07-23
経緯: 「音声で全部操作できるようにする」という発想から生まれた気づき。問題は音声認識の精度ではなく、利用者が「何を言えばいいか分からない」ことだった。

初めてFilovitaを使う人が録音ボタンを押しても、「タグ付けて？」「病院にして？」「前回と繋げて？」と自分から言える人はいない。AIは何ができるか分かっていても、人間には分からない。だからAIが少しだけ会話をリードする方が自然である。

> 「病院のお話でしたね。病院タグを付けますか？」→「うん」
> 「前回の病院の記録ともつなげられます。つなぎますか？」→「お願い」
> 「病院のホームページも登録できます。登録しますか？」

原則は3つ。

1. **質問は一つずつ。** 「タグ、リンク、マーカー、チェック、関連Event、URLを追加しますか？」と一度に聞かない。
2. **案内は使うほど後退する。** 同じ人が同じ種類のタグ(例：病院)を何度も使えば、AIはもう聞かなくなる。新しい種類のことだけ聞く。
3. **チュートリアルとして最初に教えない。** その機能を使う場面が来たときだけ、一言だけ案内する(初めてGoogle連携した時、初めてタグの道具箱を使った時、など)。

AIは操作方法を教える先生ではなく、自転車の補助輪のように「最初だけ支えて、慣れたら自然に外れる」存在である。これは音声入力に限らず、Filovita全体の案内設計の原則として扱う。「生活を邪魔しない」という思想は、機能の作り方だけでなく、機能の教え方にも及ぶ。
